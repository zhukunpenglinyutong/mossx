## Context

Issue #523 points out that Web Service remote clients currently need a new token after every restart because the settings UI starts the server without passing a token:

- frontend already calls `startWebServer({ port })`
- transport already accepts `token?: string | null`
- daemon runtime already uses `requested_token` when present and generates a random token only when absent

So the missing piece is not a new auth model; it is a small settings contract that turns “call-time optional token” into an explicit user setting.

## Design

### State Model

Add `webServiceToken` to app-level settings:

```ts
type AppSettings = {
  webServicePort: number;
  webServiceToken: string | null;
};
```

Rust mirrors the field as an optional string with `serde(default, rename = "webServiceToken")` so existing settings files load without migration.

Normalization rules:

- `null`, missing, and whitespace-only values mean “auto-generate on start”.
- non-empty trimmed values mean “fixed token”.
- save flows should not persist leading/trailing whitespace.

### UI Contract

The Web Service settings panel has two distinct token concepts:

- **Fixed token setting**: persisted user choice for future starts.
- **Runtime token**: token returned by the currently running service.

The UI should avoid merging these concepts. A running service may show a runtime token generated by daemon even when the fixed token setting is empty.

Controls:

- Fixed token input: masked/password-style by default, editable.
- Save/apply behavior: follows existing app settings update path.
- Clear action: saves `webServiceToken = null` and restores auto-generate mode.
- Generate/rotate action: creates a strong random token via Web Crypto, Rust UUID, or another existing cryptographically secure source, then saves it as the fixed token. Implementations must not use `Math.random()` or predictable pseudo-random sources.

### Start Flow

Pseudo-flow:

```text
handleStart:
  validate port
  save port if changed
  fixedToken = normalize(appSettings.webServiceToken)
  startWebServer({
    port: parsedPort,
    token: fixedToken ?? null
  })
  render returned runtime token as today
```

Important: do not write the returned runtime token back to settings when fixedToken is null. That would silently change default behavior from ephemeral to persistent.

### Security Boundary

This change does not strengthen local secret storage. It makes the current Web Service token deliberately persistent only when the user asks for that. The UI must disclose that the fixed token is saved in local app settings.

Token rotation is just replacement:

```text
old fixed token -> generate new token -> save -> next start uses new token
```

If the service is already running, saving a different fixed token does not automatically mutate runtime auth. The new token takes effect on the next start unless the implementation explicitly stops/starts as part of a user action. Phase 1 should keep this next-start boundary to avoid hidden runtime disruption.

If the service is currently stopped, “next start” means the very next Start action should use the newly saved fixed token. If the service is running, the current runtime token remains valid until stop/restart.

### Diagnostics and Redaction

`webServiceToken` is a persistent secret. Any settings diagnostics, support bundle, logs, telemetry-like summaries, or error messages must not include the raw token value.

Allowed diagnostic shape:

```json
{
  "hasWebServiceToken": true
}
```

Disallowed diagnostic shape:

```json
{
  "webServiceToken": "actual-token"
}
```

## Alternatives Considered

### Auto-persist daemon-generated token

Rejected. It solves convenience but violates the existing default random-token expectation. A user can accidentally turn a short-lived token into a durable credential without noticing.

### Keychain-backed token storage

Deferred. It is a better long-term local-secret story, but it widens the change into platform credential management and migration. This issue only needs explicit fixed-token reuse.

## Edge Cases

- Existing settings without `webServiceToken`: load as auto-generate.
- User enters only spaces: save as `null`.
- User rotates while service is running: status token remains old until restart; UI should avoid claiming immediate runtime effect.
- User clears token while service is running: current runtime token remains valid until stop; next start auto-generates.
- Existing remote backend token remains unrelated and must not be used as Web Service fixed token.
- Diagnostics bundle generation: raw `webServiceToken` must be redacted even though non-sensitive settings such as `webServicePort` remain visible.

## Validation Strategy

- Frontend unit tests:
  - start without fixed token calls `startWebServer({ port, token: null })` or equivalent null payload.
  - start with fixed token passes trimmed token.
  - clear restores auto-generate state.
  - generate/rotate saves a non-empty token from a secure random source or a wrapper that can be tested without relying on predictable output.
  - runtime token reveal/copy tests still pass.
- Type/settings tests:
  - default app settings include `webServiceToken: null`.
  - Rust settings deserialize missing field without error.
- Diagnostics tests:
  - sanitized settings include only a boolean such as `hasWebServiceToken`.
  - serialized diagnostics do not contain the raw fixed token.
- Regression:
  - Web Service status/start/stop flow remains compatible.
  - `backendMode` and `remoteBackendToken` are unchanged by fixed token edits.
